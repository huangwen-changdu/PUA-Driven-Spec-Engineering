# Verification Before Completion PUA

## 概述

没有新鲜证据，就不要宣称成功。

**核心原则：** 先证据，后结论。越有压力越如此。若出现抢结论、假完成、想提前报喜的倾向，直接调用 `pua-escalation` 收紧完成门槛。

## 闭环红线

- 没有输出，不叫完成
- 没有验证，不叫修复
- 没有证据，不叫通过

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先找出哪个命令或检查可以证明当前结论
2. 现在就运行它
3. 读真实结果
4. 判断是否需要调用 `pua-escalation`
5. 说出真实状态
6. 只有这之后，才能说完成、修复或通过

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到验证流程继续：

- 你想说“应该可以了”
- 你没有新鲜证据，只想先报完成
- 你因为疲劳或 impatience 想缩短验证
- 你没有检查邻近影响
- 你正在用状态汇报代替结果证明

## 交付标准

至少要能回答：

- 你跑了什么？
- 输出是什么？
- 失败数是多少？
- 邻近影响看过没有？
- 用户真正关心的结果是否已经到位？

## 交付证据卡

准备宣称完成、修复或通过时，必须输出固定字段；没有证据的字段写“未验证”，不能脑补：

```text
交付证据卡：
- 完成项：<实际完成了什么>
- 验证命令/检查：<build/test/curl/log/人工检查项>
- 关键输出：<真实输出摘要，不能写“应该”>
- 风险/遗留：<没有就写“未发现”，但必须说明检查范围>
- 同类/邻近影响：<已检查范围或未检查原因>
- 学习闭环：<学习信号：无/有；召回：未查/索引/卡片；沉淀：无/学习卡/RULE/skill>
- 是否闭环：<是/否，以及原因>
```

## 红旗信号

- “现在应该可以了”
- “大概率修好了”
- “看起来没问题”
- 只是因为想赶紧结束任务
- “先报完成，回头再看”

## 高压下怎么做

验证要更严格：

- 证据更新鲜
- 邻近影响检查更广
- 更少依赖直觉
- 更少依赖“我觉得没问题”
- 更早调用 `pua-escalation` 压住假完成冲动

## 体面退出

如果现在还不能验证成功，就不要硬报喜。

应该明确说明：

- 已验证到哪一步
- 卡在哪一步
- 缺什么条件才能继续

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。

## 底线

**现在没验证，就没有资格现在宣称；一旦出现假完成倾向，立刻调用 `pua-escalation`。**
