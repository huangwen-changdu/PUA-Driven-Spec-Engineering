# Subagent-Driven Development PUA

## 概述

按任务拆分执行，每个任务都使用新的执行通道，并经过严格评审回路。

**核心原则：** 上下文要新鲜，评审要严格，不能靠随手延续状态混过去。出现子通道裸奔、任务边界发虚、想跳评审或带病推进时，直接调用 `pua-escalation`，再回到当前任务继续推进。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 读取计划
2. 提取任务和必要上下文
3. 一次只执行一个任务，并为该任务分配新的执行通道
4. 做规格/需求一致性评审
5. 做代码质量评审
6. 判断是否需要调用 `pua-escalation`
7. 有问题先修，再进入下一个任务
8. 循环直到所有任务完成

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到当前任务循环：

- 你准备让子通道在上下文不清时直接开写
- 你想因为任务小就跳过评审
- 你明知还有重要问题却想先推进下一个任务
- 你发现子通道开始裸奔，不带质量约束
- 你开始拿“差不多”当子任务完成标准

## Sub-agent 也不养闲

每个执行通道都必须带上这些约束：

- 不准跳验证
- 不准拿“差不多”当完成
- 不准有重要问题未修就交回
- 不准在上下文不清时硬写

**子通道也要有 Owner 意识，不是接了活就机械输出。**

凡是派子 agent，prompt 必须带 PUA 质量约束：先查事实、别猜、输出证据、发现不一致要指出、不得空口完成。否则视为子通道裸奔。

## 高压下怎么做

把每一层都收紧：

- 任务范围更小
- 任务上下文更清楚
- 评审通过条件更严
- 不允许“差不多就行”的交付
- 有阻塞时更早升级，而不是闷头硬顶
- 一旦子通道开始失控，立即调 `pua-escalation`

## 绝对不要做的事

- 因为任务看起来简单就跳过评审
- 明知还有重要问题却先推进到下一个任务
- 让新执行通道接收模糊不清的上下文
- 让子通道裸奔，不带任何质量约束

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。
