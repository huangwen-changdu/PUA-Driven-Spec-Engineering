# C# Documentation PUA

## 概述

检查代码注释质量，确保代码有必要的文档说明，检查注释完整性、清晰度和规范性。

**核心原则：** 代码注释应该在开发过程中同步编写，不要等到最后才补充。出现想跳过注释、忽视文档质量、或只想赶进度时，直接调用 `pua-escalation`，再回到注释检查流程继续推进。

## 职责范围

**本技能专注于：**
- 注释完整性：检查是否有必要的注释
- 注释清晰度：检查注释是否清晰易懂
- 注释规范性：检查注释是否符合规范
- 文档生成：为缺少注释的代码生成合适的注释

**本技能不负责：**
- 代码质量检查（由 `code-review-pua` 负责）
- 代码简化（由 `cs-simplify-pua` 负责）
- 代码分析（由 `cs-analyze-pua` 负责）
- 功能验证（由 `function-validation-pua` 负责）

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了"有流程感"展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 确定注释检查范围
2. 检查注释完整性
3. 检查注释清晰度
4. 检查注释规范性
5. 生成注释建议
6. 判断是否需要调用 `pua-escalation`
7. 真实问题先修完，再往下走

## 检查内容

### 注释完整性
- 类注释：是否有类的功能说明
- 方法注释：是否有方法的参数、返回值、异常说明
- 复杂逻辑：是否有复杂算法或业务逻辑的说明
- 关键决策：是否有设计决策或权衡的说明

### 注释清晰度
- 语言简洁：注释是否简洁明了
- 避免废话：是否避免无意义的注释
- 更新及时：注释是否与代码同步更新
- 避免误导：注释是否准确描述代码行为

### 注释规范性
- 格式规范：是否符合项目注释格式要求
- 命名规范：注释中的术语是否一致
- 位置规范：注释是否放在合适的位置
- 语言规范：是否使用统一的语言（中文/英文）

## 注释检查方法

### 1. 类注释检查
```csharp
// 缺少注释
public class OrderService
{
    // ...
}

// 完整注释
/// <summary>
/// 订单服务类，处理订单的创建、修改、查询等业务逻辑
/// </summary>
/// <remarks>
/// 该类负责订单的全生命周期管理，包括：
/// 1. 订单创建和验证
/// 2. 库存检查和扣减
/// 3. 支付处理
/// 4. 订单状态更新
/// </remarks>
public class OrderService
{
    // ...
}
```

### 2. 方法注释检查
```csharp
// 缺少注释
public OrderResult CreateOrder(OrderRequest request)
{
    // ...
}

// 完整注释
/// <summary>
/// 创建新订单
/// </summary>
/// <param name="request">订单请求信息，包含客户ID、商品列表、收货地址等</param>
/// <returns>订单创建结果，包含订单号和状态</returns>
/// <exception cref="ArgumentNullException">当request为null时抛出</exception>
/// <exception cref="InsufficientStockException">当库存不足时抛出</exception>
/// <exception cref="InvalidAddressException">当收货地址无效时抛出</exception>
public OrderResult CreateOrder(OrderRequest request)
{
    // ...
}
```

### 3. 复杂逻辑注释检查
```csharp
// 缺少注释
if (order.Status == OrderStatus.Pending && 
    order.CreateTime < DateTime.Now.AddDays(-7) &&
    !order.HasPaid)
{
    // 复杂逻辑
}

// 完整注释
// 取消超时未支付订单的逻辑：
// 1. 订单状态为待支付
// 2. 订单创建时间超过7天
// 3. 订单未支付
// 满足以上条件的订单将被自动取消
if (order.Status == OrderStatus.Pending && 
    order.CreateTime < DateTime.Now.AddDays(-7) &&
    !order.HasPaid)
{
    // 复杂逻辑
}
```

### 4. 关键决策注释检查
```csharp
// 缺少注释
var retryCount = 3;

// 完整注释
// 重试次数设置为3次的原因：
// 1. 网络请求通常在3次重试内能够成功
// 2. 超过3次重试会显著增加系统负载
// 3. 根据历史数据，99.5%的请求在3次重试内成功
var retryCount = 3;
```

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到注释检查流程：

- 你想跳过注释检查
- 你准备带着缺失的注释往下走
- 你在淡化注释质量的重要性
- 你只想赶进度，不想补充注释
- 你打算用"后面再补"推迟注释工作

## 高压下怎么做

时间越紧，越不能缩注释检查。

正确做法是：

- 注释检查范围更聚焦
- 关键注释立即补充
- 一旦出现跳注释倾向，立即调 `pua-escalation`

## 红旗信号

- "这点东西不用注释"
- "这个后面再补"
- "先这样吧，差不多了"
- "代码自解释，不需要注释"

## 压力升级（内联保底）

优先调用 pua-escalation；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 pua-escalation 按 pua/references/methodology-router.md 切换链执行；无法加载时按内联切换动作表执行。

## 底线

代码注释的成本，永远低于维护时理解代码的成本；一旦开始想跳注释，就先调 `pua-escalation`。
