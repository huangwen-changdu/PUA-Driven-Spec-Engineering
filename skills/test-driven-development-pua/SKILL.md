# Test-Driven Development PUA

## 概述

先写失败测试，再写最小实现，再验证通过。

**核心原则：** 压力越大，TDD 越不能省。出现想先写代码、跳过失败测试、或拿“这个很小”当借口时，直接调用 `pua-escalation`，再回到 TDD 循环继续推进。

## 铁律

**没有失败测试，就不能写生产代码。**

如果代码写在测试前，删掉，回到测试起点重来。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 写一个失败测试
2. 跑起来，确认它因正确原因失败
3. 写最小实现
4. **执行代码质量检查**（调用 `code-quality-check-pua` 技能）
5. 再跑，确认测试通过
6. 判断是否需要调用 `pua-escalation`
7. 只有保持绿色时才能重构

## 批量 TDD 模式

**核心思路：铁律不变（没有失败测试就不能写生产代码），但把"逐任务循环"改为"分阶段批量"。**

### 标准模式 vs 批量模式

| | 标准 TDD | 批量 TDD |
|---|---------|---------|
| 流程 | 任务A: 写测试→跑→实现→跑 → 任务B: 写测试→跑→实现→跑 | 批量: 写A+B+C测试 → 跑全部 → 实现A+B+C → 跑全部 |
| 适用 | R3+ 高风险、任务间无关联、复杂度高 | R0-R2、任务粒度小、同模块/同文件区域 |
| 步骤数（5任务） | 20 步 | ~6 步 |
| 验证粒度 | 每个任务独立验证 | 批量验证，失败时逐个排查 |

### 触发条件

**自动进入批量模式**（同时满足）：
- 来源是已确认的 tasks.md 计划
- 当前批次有 2+ 个连续任务
- 任务风险等级 ≤ R2
- 任务在同一模块/文件区域，或有明确依赖链

**强制标准模式**（任一命中）：
- R3+ 高风险任务
- 任务间完全无关联（跨模块/跨服务）
- 用户明确要求逐任务 TDD
- 前一批次批量验证失败且原因不明

### 批量流程

```
阶段 1：批量写测试
  ├─ 为当前批次（3-5 个任务）一次性写完所有失败测试
  ├─ 跑全部测试，确认全部因正确原因失败
  └─ 输出：{N} 个测试全部 FAIL ✓

阶段 2：批量实现
  ├─ 一次性实现这批任务的生产代码
  └─ 注意：仍然遵循"最小实现"原则，不做多余的事

阶段 3：批量验证
  ├─ 跑全部测试，确认全部通过
  ├─ 全绿 → 进入下一批次或重构
  └─ 有红 → 定位失败任务，修复后重跑（不回退到阶段 1）
```

### 批次大小建议

| 任务复杂度 | 建议批次大小 | 说明 |
|-----------|------------|------|
| 单函数/单字段修改 | 5 个 | 粒度小，批量收益最大 |
| 单文件多处修改 | 3-4 个 | 同文件内改动，上下文共享 |
| 跨文件但同模块 | 2-3 个 | 需要切换文件，批次不宜过大 |
| 跨模块 | 1 个（标准模式） | 上下文差异大，不适合批量 |

### 批量模式下的失败处理

```
批量验证有红：
  1. 定位具体失败的测试（不全部重来）
  2. 分析失败原因：
     - 实现 bug → 修复实现，重跑
     - 测试写错 → 修复测试，重跑
     - 依赖冲突 → 拆出该任务，单独走标准 TDD
  3. 修复后重跑全部（确认无回归）
  4. 连续 2+ 次批量失败 → 回退到标准模式
```

### 铁律不变

批量模式**不放松**以下约束：
- **没有失败测试，就不能写生产代码**（只是批量写测试，不是跳过测试）
- **必须确认测试因正确原因失败**（只是批量确认，不是不确认）
- **必须验证通过才能继续**（只是批量验证，不是不验证）

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到当前 TDD 步骤：

- 你想先快速补一下代码
- 你觉得这个修复太小，不值得先写测试
- 你准备把测试后补
- 你开始拿“我已经知道怎么改了”替代失败测试
- 你准备带着没验证的实现继续往前走

## 红旗信号

- “我先快速补一下”
- “这个修复我已经知道了”
- “测试后补就行”
- “这个太小，不值得 TDD”

## 高压下怎么做

- 测试范围更小
- 实现更小
- 循环更短
- 但一步不能跳
- 一旦想跳步骤，立即调 `pua-escalation`

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。

## 底线

**高压下最快的安全路径，依然是 红 → 绿 → 重构；一旦想跳过失败测试，就先调 `pua-escalation`。**
