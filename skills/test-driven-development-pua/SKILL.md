# Test-Driven Development PUA

## 概述

本 skill 的目标不是“每个功能/每个 Bug 都强制跑完整 TDD 仪式”，而是用**最少 token**拿到足够的行为保护证据。

**核心原则：** 默认轻量、按风险升级。普通改动用最小验证闭环；高风险、复杂行为、连续失败时才进入严格红绿循环。

## 触发边界

### 进入本 skill

仅在同时满足以下条件时进入：

1. 已完成需求/根因/计划确认，准备改生产代码。
2. 本次改动会改变可观察行为，或需要测试证据防回归。
3. 当前仓库存在可运行的相关测试，或新增一个小测试的收益明显高于成本。

### 不进入本 skill

以下场景不要为了“流程完整”加载 TDD：

- 还在排查 Bug、定位根因、复现问题：先走 `systematic-debugging-pua`。
- 只改文档、规则、提示词、注释、文案、格式化或配置说明。
- 机械性小改，已有 lint/build/typecheck 足以验证。
- 目标项目没有相关测试入口，且补测试需要大规模搭架子。
- 执行已确认计划时的每个小步骤：继承计划门禁，批量验证即可。

## 模式选择

| 模式 | 默认级别 | 适用场景 | 要求 |
|---|---|---|---|
| **TDD-Lite** | 默认 | R0-R1、小范围行为改动、单文件/同模块修改 | 定义成功标准 → 最小实现 → 跑最相关验证；有现成测试才先红后绿 |
| **Regression Test** | Bug 修复 | 根因已确认且容易写复现用例 | 只补 1 个最小回归测试；不为写测试重构大块代码 |
| **Batch TDD** | 计划执行 | R0-R2、同模块、多个小任务 | 批量写/调整测试 → 批量实现 → 批量验证 |
| **Strict TDD** | 升级模式 | R2+、资金/权限/数据/核心流程、复杂算法、连续失败 | 必须先失败测试，再最小实现，再通过验证 |

## 默认轻量流程

0. **门禁轻量化**：进入本 skill 前仍按总规则完成本轮 `pua-gate`；进入已确认计划的执行批次后，不在每个微任务重复展开长门禁，只有发现新风险、验证失败或需求变了才重新完整门禁。
1. **选验证策略**：优先复用现有测试；没有合适测试时，用 lint/build/typecheck/手工可复现步骤替代。
2. **最小改动**：只改本次目标需要的代码，不顺手重构。
3. **目标验证**：只跑最相关命令；最后收尾前再跑更宽验证。
4. **失败处理**：验证失败先定位原因；同一问题连续 2 次失败再切 `systematic-debugging-pua` 或 `pua-escalation`。

## 什么时候必须先写失败测试

命中任一条件，升级为 Strict TDD：

- R2+ 风险：数据持久化、权限认证、资金权益、核心流程、跨服务契约。
- Bug 根因明确，且能用一个小用例稳定复现。
- 新增复杂分支、算法、状态机或边界条件。
- 同类问题之前修过又回来了。
- 用户明确要求 TDD 或回归测试。

## 什么时候允许不新增测试

满足以下任一条件，可以只做目标验证，但必须说明验证证据：

- 改动不改变运行行为：文档、规则、注释、格式、文案。
- 已有测试覆盖该路径，只需运行相关测试。
- 没有测试基础设施，补测试会显著扩大范围。
- 修复点只能通过构建、类型检查、lint、快照或手工步骤验证。
- 本次目标是先止血，后续测试另列任务并明确风险。

## 批量 TDD 模式

**核心思路：** 铁律按批次生效，不按每个微任务重复消耗上下文。

### 触发条件

同时满足：

- 来源是已确认计划或同一批相关改动。
- 当前批次有 2+ 个连续小任务。
- 风险等级 ≤ R2。
- 任务在同一模块/文件区域，或有明确依赖链。

### 批量流程

```text
阶段 1：批量准备验证
  ├─ 复用或补充当前批次需要的最小测试/验证点
  └─ 如有新测试，确认它能暴露目标问题或新行为

阶段 2：批量实现
  ├─ 一次性完成当前批次的最小生产代码改动
  └─ 不顺手重构，不扩大范围

阶段 3：批量验证
  ├─ 跑相关测试/构建/lint/typecheck
  ├─ 全绿 → 进入下一批次或收尾验证
  └─ 有红 → 定位失败项，只拆出失败项单独处理
```

## 代码质量检查策略

不要在 TDD 每一步强制调用 `code-quality-check-pua`。默认规则：

- 小改动：目标验证通过即可。
- 跨文件、R2+、公共 API、核心流程：收尾前调用 review/quality/verification。
- 验证失败或 diff 变大：再升级到 `code-quality-check-pua` 或 `requesting-code-review-pua`。

## 输出预算

普通执行只输出三类信息：

1. 选择的模式：`TDD-Lite` / `Regression Test` / `Batch TDD` / `Strict TDD`。
2. 改了什么：文件和行为点。
3. 证据：命令、真实输出、判定。

禁止输出长篇 TDD 教程、重复门禁表、逐微任务状态播报。

## 红旗信号

出现以下情况才升级，而不是每次都升级：

- 想跳过所有验证直接报完成。
- 高风险改动没有任何测试或回滚证据。
- 同一验证连续失败 2 次还在猜。
- 生产代码越改越大，偏离原目标。

## 底线

**TDD 是证据手段，不是 token 仪式。默认轻量闭环；风险升高、失败重复、行为复杂时，再升级严格红绿循环。**
