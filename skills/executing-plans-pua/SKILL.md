# Executing Plans PUA

## 概述

按计划顺序执行，真的遇到阻塞时就停下来，不要硬顶。

**核心原则：** 计划就是执行通道；再急，也不能绕过计划自由发挥。出现擅自改意图、跳验证、带着阻塞硬推进的倾向时，直接调用 `pua-escalation`，再回到当前任务继续执行。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先完整阅读并质疑这份计划
2. 有关键缺口先提出来
3. 一次执行一个任务
4. 按计划要求验证每个任务
5. **执行代码质量检查**（调用 `code-quality-check-pua` 技能）
6. 判断是否需要调用 `pua-escalation`
7. 有未解决阻塞时不要强行往前滚动

## 执行阶段门禁降级协议

**核心思路：计划已确认 = 门禁已前置。执行阶段不需要每步重评门禁。**

计划经用户确认进入执行阶段后，任务执行循环中的门禁从"每轮必须完整重评"降为**异常触发模式**：

### 正常执行（降级模式）

每个任务执行步骤只需输出一行轻标，不做完整门禁拆解：

```text
🟠 执行中 · 任务 {N}/{M} · {一句话描述}
```

不触发完整 pua-gate 的条件：
- 执行动作在计划范围内
- 测试通过或失败原因明确
- 未发现新风险或新依赖

### 异常触发完整门禁

出现以下**任一**情况，立即恢复完整 pua-gate：

| 触发信号 | 说明 |
|---------|------|
| 执行偏离计划 | 实际改动超出计划任务的范围/意图 |
| 测试连续失败 2+ 次 | 同一任务的验证反复不通过 |
| 发现新依赖/新风险 | 代码搜索发现计划未覆盖的上下游影响 |
| 用户主动 `/pua` | 用户明确要求收紧流程 |
| 假完成信号 | 说"完成"但验证命令未跑或输出不符 |

### 为什么可以降级

1. **门禁已前置**：计划阶段已经过 brainstorming-pua / pua-gate 的完整审查
2. **任务已拆解**：tasks.md 中每个任务的范围、验证方式已明确
3. **风险已评估**：变更风险等级在计划阶段已确定
4. **执行阶段的核心风险是"偏离计划"**，不是"需求不清"——用异常触发比每轮重评更精准

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到当前执行步骤：

- 你准备悄悄改计划意图
- 你因为赶时间想跳过某个验证
- 你明知阻塞存在还想硬往前滚
- 你开始用“差不多能跑”代替任务完成
- 你想靠口头解释覆盖计划缺口

## 高压下怎么做

- 执行通道更窄
- 验证更频繁
- 阻塞升级更早
- 不要悄悄改计划意图
- 一旦开始失控，立即调 `pua-escalation`

## 停止条件

出现以下情况就该停并上报：

- 计划缺少关键细节
- 阻塞已经影响正确执行
- 验证连续失败

## 压力升级

当遇到困难或失败时，读取 `skills/pua-escalation/SKILL.md` 获取具体的解决方案和指导。

## 底线

单通道执行能成立的前提，是这条通道始终受控；一旦开始脱轨，就先调 `pua-escalation` 收紧，再继续执行。
