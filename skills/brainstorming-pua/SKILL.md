# Brainstorming PUA（OpenSpec SDD 模式）

## 概述

先把想法变成**结构化的规范文档**，再谈实现。

**核心原则：** 压力可以压缩表达，但不能压缩设计纪律。采用 **OpenSpec 规范驱动开发（SDD）** 框架——做一步确认一步，前一层未确认，后一层无法正确生成。

**"变更即代码"**：变更不是对话中的想法，而是可追踪、可版本化、可归档的结构化文档资产。每个变更在 `openspec/changes/{change-name}/` 下有独立的四层产物。

出现"先写再说""先计划再补设计""先动起来别想太多"的冲动时，直接调用 `pua-escalation`，再回到设计阶段继续收敛。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了"有流程感"展开长门禁。

## OpenSpec 四层渐进流程

**核心机制：做一步确认一步。** 每层产物完成后必须经用户确认，才能进入下一层。AI 不会在规范未明确时自行发挥。

```
Proposal ─────→ Specs ─────→ Design ─────→ Tasks
(确认)          (确认)       (确认)        (确认) → writing-plans-pua
```

四层依赖链：`proposal → specs + design → tasks → apply`。前一层未确认，后一层无法正确生成。

### 第一层：Proposal（提案）

**回答的问题**：为什么改？改什么？影响谁？

**产出**：`openspec/changes/{change-name}/proposal.md`

**格式**：

```markdown
# {变更名称}提案

## Why
[为什么需要这个变更——背景与动机]

## What Changes
- [具体变更内容列表]

## Capabilities
### New Capabilities
- `{capability-name}`: [能力描述]

### Modified Capabilities
- [已有的受影响能力]

## Impact
- 后端：[影响的模块/类/表]
- 数据：[涉及的持久化变更]
- 依赖：[新增的外部依赖或接口变更]

## Risk Assessment
- 变更风险等级：R{0-4}
- 影响文件数：{N}
- 是否命中复合升级：{是/否}
```

**确认门控**：
- 用户确认"要做这些、不做那些"后才进入下一步
- 如果用户对范围有异议，修改 proposal 后重新确认
- 复合升级命中时，proposal 必须包含 Risk Assessment 章节

### Proposal 确认后：路径选择门控

**Proposal 确认后，AI 必须暂停并向用户提出路径选择，不得自动推进到 Specs。**

输出以下选项让用户选择：

```
📋 Proposal 已确认。请选择后续推进路径：

A. 完整 Specs 路径（推荐高风险 / 需求边界模糊时）
   → 逐项澄清需求边界、验收标准、非目标，确保实现前规范清晰

B. writing-plans 快速路径（适合需求已清晰的低风险变更）
   → 跳过 Specs，直接进入 writing-plans-pua 生成实现计划

请选择 A 或 B：
```

- 用户选择 **A**：继续进入第二层 Specs，逐项深度澄清核心细节
- 用户选择 **B**：退出 brainstorming-pua，直接进入 `writing-plans-pua`
- 用户未明确选择时：等待用户响应，不得自行推进
- **R3+ 风险时不得提供 B 选项**：高风险变更必须走完整 Specs + Design 流程

### 第二层：Specs（需求规范）

**回答的问题**：需求边界？验收标准？什么不做？

**产出**：`openspec/changes/{change-name}/specs/spec.md`

**格式**：

```markdown
# {能力名称}需求规范

## 概述
[能力的高层次描述]

## 功能需求
### {维度1}
- [具体需求点]

### {维度2}
- [具体需求点]

## 验收标准
- [ ] [可验证的验收条件1]
- [ ] [可验证的验收条件2]

## 非目标
- 不在此次实现：[明确排除的范围]

## 边界条件
- [边界情况1]：[处理方式]
- [边界情况2]：[处理方式]
```

**确认门控**：
- 用户确认"需求边界和验收标准"后才进入下一步
- **非目标章节非常重要**，它明确划定边界，防止越界实现
- 验收标准必须是可验证的，不能是"功能正常"这种模糊描述

**澄清队列**：如果 proposal 阶段仍有未澄清的模糊点，在 specs 阶段继续逐项澄清（遵循 pua-gate 的多轮澄清协议）。specs 确认 = 澄清队列清空。

### 第三层：Design（技术设计）

**回答的问题**：怎么做？接口定义？关键决策？回滚方案？

**产出**：`openspec/changes/{change-name}/design.md`

**格式**：

```markdown
# {变更名称}技术设计

## 方案对比
### 方案 A：{名称}
- 描述：[方案说明]
- 优点：[...]
- 缺点：[...]
- 影响面：[...]

### 方案 B：{名称}
- 描述：[方案说明]
- 优点：[...]
- 缺点：[...]
- 影响面：[...]

### 推荐：{方案} 及原因
[推荐理由]

## 架构
- [模块/层次划分]
- [数据流向]

## 关键决策
- [决策1]：[原因]
- [决策2]：[原因]

## 接口定义
- [HTTP 方法/函数签名] [路径/参数]
- [输入输出格式]

## 变更清单
- [ ] [具体代码变更1]
- [ ] [具体代码变更2]

## 回滚方案（R3+ 必填）
### 回滚步骤
1. [步骤1]
2. [步骤2]

### 不可逆操作
- [操作]：[为什么不可逆]

### 回滚验证清单
- [ ] [验证项1]
- [ ] [验证项2]

## 影响面分析（R4 必填，R3 建议填）
### 上下游依赖
- 上游：[依赖方1] → [影响说明]
- 下游：[消费方1] → [影响说明]

### 数据流向
- [数据流向描述]

### 受影响模块清单
| 模块 | 变更类型 | 影响说明 |
|---|---|---|
| [模块名] | [新增/修改/删除] | [说明] |
```

**确认门控**：
- 用户确认"技术方案和接口定义"后才进入下一步
- R2+：方案对比必须包含至少 2 个方案
- R3+：回滚方案必须单独确认
- R4：影响面分析必须单独确认
- **复合升级命中时**：design 必须包含方案对比 + 回滚方案 + 影响面分析，三项全部确认

### 第四层：Tasks（实现任务清单）

**回答的问题**：实现步骤？任务粒度？执行顺序？

**产出**：`openspec/changes/{change-name}/tasks.md`

**格式**：

```markdown
# {变更名称}实现任务

## 前置条件
- [ ] [前置条件1]
- [ ] [前置条件2]

## 任务清单
- [ ] 1. [任务描述]（预估：{时间}）
  - 涉及文件：[文件列表]
  - 验证方式：[如何验证]
- [ ] 2. [任务描述]（预估：{时间}）
  - 涉及文件：[文件列表]
  - 验证方式：[如何验证]

## 依赖关系
- 任务2 依赖 任务1
- 任务3 可与 任务2 并行

## 验证计划
- [ ] [集成验证步骤1]
- [ ] [集成验证步骤2]
```

**确认门控**：
- 用户确认"任务拆解和执行顺序"后才进入实现
- 每个任务粒度控制在可审查的范围内
- 复合升级命中时，验证计划必须包含回滚验证

## 流程执行规则

### 做一步确认一步

**每个阶段完成后必须暂停，呈现给用户确认，确认后才进入下一阶段。** 这是 OpenSpec 的核心纪律：

1. 完成 proposal → 呈现给用户 → 用户确认 → 进入 specs
2. 完成 specs → 呈现给用户 → 用户确认 → 进入 design
3. 完成 design → 呈现给用户 → 用户确认 → 进入 tasks
4. 完成 tasks → 呈现给用户 → 用户确认 → 进入 writing-plans-pua

**不允许跳步**：不能一次输出全部四层文档然后问"这样可以吗"。每一层都是对上一层的细化与确认，前一层未完成则后一层无法正确生成。

### 需求成熟度与 OpenSpec 流程的关系

| 成熟度 | 进入 OpenSpec 的深度 | 说明 |
|---|---|---|
| 0-4 | 必须走完整四层 + 多轮澄清 | 需求严重不足，每层都需要充分讨论 |
| 5-6 | 必须走完整四层 | 需要设计澄清，但每层可以更快 |
| 7-8 | Proposal 可内联确认 + Specs/Design/Tasks 正常走 | 需求基本清晰，proposal 可快速确认 |
| 9-10 | 可简化但不可跳过 | 即使需求完全清晰，R2+ 也必须走完整流程 |

### 变更风险与 OpenSpec 流程的关系

| 风险等级 | OpenSpec 流程要求 |
|---|---|
| R0-R1 | 无强制要求（可简化或跳过 OpenSpec） |
| R2 | 必须走完整四层 + proposal 含 Risk Assessment |
| R3 | 必须走完整四层 + design 含回滚方案章节 |
| R4 | 必须走完整四层 + design 含回滚方案 + 影响面分析章节 |

### 文档落地规则

**R2+ 变更的 OpenSpec 文档必须落地到 `openspec/changes/{change-name}/` 目录。** 对话是过程，文档是结果。

- R2+：proposal.md + specs/spec.md + design.md + tasks.md 全部落地
- R3+：design.md 内含回滚方案章节
- R4：design.md 内含回滚方案 + 影响面分析章节

**没有文档不允许进入 writing-plans-pua。**

### 归档

变更完成并验证后，执行归档：

```
openspec/changes/{change-name}/ → openspec/changes/archive/{date}-{change-name}/
```

归档时更新项目主规范，确保已完成的变更反映到项目整体规范中。

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到当前设计流程继续：

- 你开始想跳过设计直接实现
- 你想把计划当设计来糊弄过去
- 用户催压导致你想减少关键澄清
- 你只盯当前点，不想补边界和同类问题
- 你已经在用模糊措辞掩盖设计没定下来
- 你想一次输出多层 OpenSpec 文档跳过逐步确认
- 你在某个阶段没有获得用户确认就自动进入下一阶段

## P8 视角

设计时必须额外想两件事：

- **还有什么没想到？**
- **还有什么类似的地方也要一起解决？**

不能只盯眼前这个点。

## 硬门禁

在设计方向明确并被确认前：

- 不写代码
- 不进入实现
- 不直接写计划冒充设计完成
- 不跳过 OpenSpec 的逐步确认
- 不一次输出多层文档跳过中间确认

## 高压下怎么做

如果用户很急：

- 少废话
- 选项更锋利
- 边界更清楚
- 结论更直接
- 每层确认更快但**不能省略**
- 一旦有跳设计倾向，立即调 `pua-escalation`
- 可以简化表达，不能简化步骤

## 常见失败

| 失败方式 | 修正方式 |
|---|---|
| 直接跳到代码 | 留在设计阶段 |
| 一次问太多问题 | 一次只问一个 |
| 只给一个方案 | 至少给可比较方案 |
| 只看当前点，不看相邻影响 | 明确补上边界和同类问题分析 |
| 把模糊需求当成已确认设计 | 明确要确认 |
| 高风险变更方案只停留在对话中 | 必须落地文档，没有文档不允许退出 |
| 一次输出全部四层文档 | 做一步确认一步，每层单独确认 |
| 某层没确认就进入下一层 | 回到未确认层，先获得确认 |
| 把 OpenSpec 当负担而非保护 | OpenSpec 是让高风险变更"零返工"的机制 |

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|---------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。
