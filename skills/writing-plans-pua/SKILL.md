# Writing Plans PUA

## 概述

把需求写成无需猜测就能执行的计划。

**核心原则：** 模糊计划带来的是假速度和真返工。计划质量下滑时，不要硬进执行，直接调用 `pua-escalation` 把计划重新收紧。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先梳理文件和边界
2. 把工作拆成可执行任务
3. 写出精确路径、明确步骤、验证方式
4. 补齐上下游影响和同类问题扫描
5. 清除占位符
6. 判断是否需要调用 `pua-escalation` 收紧计划
7. 选择执行路径：
   - `subagent-driven-development-pua`
   - `dispatching-parallel-agents-pua`
   - `executing-plans-pua`

## 计划标准

每个任务至少要说明：

- 改什么
- 在哪改
- 怎么测
- 如何判断成功
- 对上下游有什么影响
- 有没有同类位置要一起处理

## 直接调用 `pua-escalation` 的触发条件

若出现任一情况，先调用 `pua-escalation`，再回到计划阶段继续补齐：

- 计划里出现空话、占位词、`TODO`、`TBD`
- 没写验证路径，只写“完成后检查”
- 没有影响分析
- 没有同类问题扫描
- 想用“先做再说”跳过计划精化

## L3+ 强制检查

如果已经多次失败或压力明显升高，计划必须额外检查：

- 有没有隐藏假设
- 有没有遗漏边界条件
- 有没有把验证写成空话
- 有没有把风险甩给执行阶段再想

## 高压下怎么做

越急，计划越要具体：

- 范围更紧
- 任务更小
- 验证更清楚
- 少把关键假设藏在描述里
- 不允许“先做再说”式规划
- 发现计划发虚时，立即调用 `pua-escalation`

## 红旗信号

- `TODO` / `TBD`
- “按需实现”
- “处理边界情况”但没写清楚
- 没有测试或验证路径
- 没有影响分析
- 没有说明类似问题是否要一起处理

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。

## 底线

只有执行者不需要自由发挥，这份计划才算合格；只有计划能支撑闭环交付，这份计划才算过线；计划一发虚，就先调 `pua-escalation` 收紧，再谈执行。
