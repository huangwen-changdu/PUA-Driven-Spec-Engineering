# Writing Plans PUA

## 概述

把需求写成无需猜测就能执行的计划。

**核心原则：** 模糊计划带来的是假速度和真返工。计划质量下滑时，不要硬进执行，直接调用 `pua-escalation` 把计划重新收紧。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先梳理文件和边界
2. 检查是否存在 `Feature Ledger`：确认本次 `Feature`、`Iteration`、历史约束和历史修复点是否已进入计划输入
3. 把工作拆成可执行任务
4. 写出精确路径、明确步骤、验证方式
5. 补齐上下游影响、同类问题扫描和历史回归点
6. 清除占位符
7. 判断是否需要调用 `pua-escalation` 收紧计划
8. 选择执行路径：
   - `subagent-driven-development-pua`
   - `dispatching-parallel-agents-pua`
   - `executing-plans-pua`

## 计划标准

每个任务至少要说明：

- 改什么
- 在哪改
- 怎么测
- 如何判断成功
- 对上下游有什么影响
- 有没有同类位置要一起处理
- 如果本次属于已有功能迭代：所属 `Feature`、本次 `Iteration`、引用的 `Feature Ledger` 路径是什么
- 是否受功能档案中的已知约束、关键决策或历史修复点影响
- 验证计划是否覆盖历史修复点，避免同一功能回归

## 直接调用 `pua-escalation` 的触发条件

若出现任一情况，先调用 `pua-escalation`，再回到计划阶段继续补齐：

- 计划里出现空话、占位词、`TODO`、`TBD`
- 没写验证路径，只写“完成后检查”
- 没有影响分析
- 没有同类问题扫描
- 想用“先做再说”跳过计划精化

## L3+ 强制检查

如果已经多次失败或压力明显升高，计划必须额外检查：

- 有没有隐藏假设
- 有没有遗漏边界条件
- 有没有把验证写成空话
- 有没有把风险甩给执行阶段再想

## 高压下怎么做

越急，计划越要具体：

- 范围更紧
- 任务更小
- 验证更清楚
- 少把关键假设藏在描述里
- 不允许“先做再说”式规划
- 发现计划发虚时，立即调用 `pua-escalation`

## 红旗信号

- `TODO` / `TBD`
- “按需实现”
- “处理边界情况”但没写清楚
- 没有测试或验证路径
- 没有影响分析
- 没有说明类似问题是否要一起处理

## 压力升级

当遇到困难或失败时，读取 `skills/pua-escalation/SKILL.md` 获取具体的解决方案和指导。

## 底线

只有执行者不需要自由发挥，这份计划才算合格；只有计划能支撑闭环交付，这份计划才算过线；计划一发虚，就先调 `pua-escalation` 收紧，再谈执行。
