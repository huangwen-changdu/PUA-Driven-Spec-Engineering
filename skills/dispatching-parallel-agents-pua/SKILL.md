# Dispatching Parallel Agents PUA

## 概述

只有在任务真正独立时，才使用并行通道。

**核心原则：** 并行是为了吞吐，不是为了制造混乱。出现边界模糊、责任不清、用并行掩盖认知不清时，直接调用 `pua-escalation`，再回到并行调度继续收敛。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先确认任务彼此独立
2. 每组独立任务分配一个通道
3. 明确边界和责任归属
4. 协调发现、阻塞和依赖
5. 判断是否需要调用 `pua-escalation`
6. 通过评审和验证把结果重新收拢

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到并行调度流程：

- 任务其实不独立，但你还想硬拆并行
- 多个通道开始改同一批文件却没人协调
- 责任边界已经说不清
- 你想用并行掩盖范围不明或理解不足
- 合流前就想先报进度或报完成

## 子 agent 约束

凡是派子 agent，prompt 必须带 PUA 质量约束：先查事实、别猜、输出证据、发现不一致要指出、不得空口完成。否则视为子通道裸奔。

## 高压下怎么做

不要把并行当成跳过协调的借口。

在高压下要加强：

- 边界清晰度
- 通道责任归属
- 合流时的评审强度
- 问题一暴露就更早调 `pua-escalation`

## 红旗信号

- 多个通道改同一批文件却没人协调
- 责任不清
- 用并行掩盖认知不清或范围不明

## 压力升级

当遇到困难或失败时，读取 `skills/pua-escalation/SKILL.md` 获取具体的解决方案和指导。

## 底线

只有在独立性真实、合流纪律足够强时，并行才有价值；一旦并行开始制造混乱，就先调 `pua-escalation` 收紧协调。
