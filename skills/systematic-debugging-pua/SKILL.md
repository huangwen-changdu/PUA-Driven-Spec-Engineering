# Systematic Debugging PUA

## 概述

靠证据调试，不靠慌乱调试。

**核心原则：** 没有根因调查，就不要动修复。出现连续失败、乱补丁、漂移时，直接调用 `pua-escalation` 升级压力与证据要求。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 读真实报错和输出
2. 稳定复现
3. 检查最近改动和边界
4. 对比损坏路径和正常样例
5. 提出一个明确假设
6. 只测试一个改动
7. 判断是否需要调用 `pua-escalation`
8. 验证结果
9. 修完后做同类问题复盘

## 深度 PUA 通用方法论

卡壳时必须按这个顺序走：

1. **闻味道**：你是不是在重复同一种思路？
2. **揪头发**：逐字读信号、主动搜索、读原始上下文、验证假设、反转假设
3. **照镜子**：是不是该查没查、该换没换、该停没停？
4. **执行新方案**：必须和之前本质不同
5. **复盘**：检查同类问题、缺口和预防动作

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到当前调试循环继续：

- 两次相似尝试已经失败
- 你准备堆叠多个猜测性补丁
- 你还没读懂信号就想先改代码
- 你开始把“环境问题”当默认借口
- 你开始用“建议用户手动处理”提前退出

## 压力升级

当遇到困难或失败时，读取 `skills/pua-escalation/SKILL.md` 获取具体的解决方案和指导。

## 硬门禁

**不要堆叠猜测性修复。**

如果两次相似尝试都失败了，第三次必须换假设，不是换运气。

## 7 项检查清单（E3+ 强制）

- [ ] 逐字读完失败信号了吗？
- [ ] 搜索过核心问题了吗？
- [ ] 读过失败位置的原始上下文了吗？
- [ ] 所有关键假设都验证了吗？
- [ ] 试过相反假设了吗？
- [ ] 能最小复现吗？
- [ ] 换过工具/方法/角度了吗？

## 红旗信号

- “先来个临时补丁”
- “一次试几个改动看看”
- “我还没看懂，但这个可能有用”
- “建议用户手动处理”但你其实还没穷尽方法
- “可能是环境问题”但没有验证

## 底线

**压力越大，调试越要更深，不是更松；一旦进入重复失败或乱补丁倾向，立刻调用 `pua-escalation`；没穷尽一切，就不要轻易放弃。**
