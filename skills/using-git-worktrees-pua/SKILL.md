# 使用 Git Worktrees PUA

## 概述

实现前先建立隔离工作区。

**核心原则：** 再急，也不能在错误的位置开干。出现想图省事原地改、跳过基线检查、或带着脏环境直接开工的倾向时，直接调用 `pua-escalation`，再回到隔离准备流程继续推进。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 按已有规则或偏好决定 worktree 位置
2. 验证项目内 worktree 目录已被忽略
3. 创建 worktree
4. 运行项目初始化
5. 校验干净基线
6. 判断是否需要调用 `pua-escalation`
7. 报告可用路径

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到 worktree 准备流程：

- 你想直接在主工作区开改
- 你想跳过基线检查
- 你想先改一点再说，之后再隔离
- 你准备带着未确认的脏状态继续实现
- 你想拿“改动很小”当不做隔离的理由

## 硬门禁

如果当前工作应当隔离：

**不要在主工作区直接开始实现。**

## 高压下禁止自我合理化

- “我先在这改一下，等会再隔离”
- “这个改动很小，不用 worktree”
- “基线待会再测”
- “先开干再整理环境”

这些都是脏状态和假通过的来源；一旦出现，就先调 `pua-escalation`。

## 失败规则

如果基线本来就是坏的，先报告，再决定是否继续。

## 压力升级（内联保底）

优先调用 `pua-escalation`；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 `pua-escalation` 按 `pua/references/methodology-router.md` 切换链执行；无法加载时按内联切换动作表执行。

## 交接

工作区准备好后，进入 `writing-plans-pua` 或已确定的执行技能。
