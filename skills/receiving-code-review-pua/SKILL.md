# Receiving Code Review PUA

## 概述

用技术方式处理反馈，不要演戏。

**核心原则：** 先验证，再修复；不对的就带着证据顶回去。若开始表演式认同、机械照改、或反馈压力很高，直接调用 `pua-escalation` 收紧闭环。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 先把全部反馈读完
2. 任何不清楚的项，先澄清，再实现
3. 每条反馈都对照代码事实验证
4. 判断是否需要调用 `pua-escalation`
5. 有效问题一次修一个
6. 需要时重新发起评审

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到 feedback 处理流程：

- 你已经开始机械照单全收
- 你想用“谢谢”“你说得对”代替技术判断
- 反馈很多，开始只想尽快过关
- 你准备改动自己都没验证过的建议
- 你发现自己在拿态度代替闭环

## 绝对不要做的事

- 表演式认同
- 用“谢谢”“你说得对”代替行动
- 把不清楚和清楚的问题混着一起改
- 没验证就直接执行存疑建议

## 高压下怎么做

压力越大，处理反馈越应该：

- 更短
- 更清楚
- 更技术化
- 更少情绪表达
- 更早调用 `pua-escalation` 收紧标准

## 底线

对的就修，错的就顶，谁也别糊弄过去；一旦开始演、开始糊、开始只想过关，就先调 `pua-escalation`。
