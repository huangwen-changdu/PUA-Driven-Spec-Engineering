# Code Quality Check PUA

## 概述

在开发过程中自动进行代码质量检查，集成代码审查、代码简化、代码分析和功能验证。

**核心原则：** 代码质量检查应该在开发过程中自动进行，而不是事后手动触发。出现想跳过质量检查、淡化质量问题、或只想赶进度时，直接读取 `skills/pua-escalation/SKILL.md` 获取解决方案，再回到质量检查流程继续推进。

## 集成的检查功能

本技能集成了以下五个代码质量检查功能：

1. **code-review**：代码审查，检查代码质量、发现潜在问题、优化代码性能、检查安全漏洞
2. **cs-doc**：代码注释检查，检查注释完整性、清晰度和规范性
3. **cs-simplify**：C# 代码简化，简化和优化 C# 代码
4. **cs-analyze**：C# 代码分析，分析 C# 代码质量，找出潜在问题和改进建议
5. **function-validation**：功能验证，验证代码功能是否正确实现、检查是否符合最佳实践、评估代码质量

## 职责划分（避免重复检查）

**各技能职责边界清晰，避免重复检查：**

| 技能 | 主要职责 | 检查重点 | 不负责 |
|------|----------|----------|--------|
| **code-review** | 代码质量审查 | 可读性、可维护性、可扩展性、潜在问题、性能、安全 | 详细注释检查（由cs-doc负责） |
| **cs-doc** | 注释质量检查 | 注释完整性、清晰度、规范性 | 代码质量、性能、安全 |
| **cs-simplify** | 代码简化 | 代码冗余、复杂度、重构机会 | 注释质量、功能验证 |
| **cs-analyze** | 代码分析 | 命名规范、内存泄漏、资源释放、线程安全 | 代码简化、功能验证 |
| **function-validation** | 功能验证 | 功能正确性、最佳实践、代码质量 | 代码简化、注释质量 |

**协作关系：**
- `code-review` 发现注释问题时，建议调用 `cs-doc` 进行详细检查
- `cs-simplify` 和 `cs-analyze` 各自专注不同维度，不重复检查
- `function-validation` 专注功能正确性，不重复其他技能的检查内容

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即读取 `skills/pua-escalation/SKILL.md` 获取解决方案，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了"有流程感"展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 确定检查范围（单个文件、多个文件、整个模块）
2. 执行代码审查（code-review）
3. 执行注释检查（cs-doc）
4. 执行代码简化（cs-simplify）
5. 执行代码分析（cs-analyze）
6. 执行功能验证（function-validation）
7. 汇总检查结果，区分严重、重要、次要问题
8. 判断是否需要读取 `skills/pua-escalation/SKILL.md` 获取解决方案
9. 真实问题先修完，再往下走

## 检查流程详解

### 1. 代码审查（code-review）

**检查内容：**
- 代码质量：可读性、可维护性、可扩展性
- 潜在问题：逻辑错误、边界条件、异常处理
- 性能优化：算法效率、资源使用、缓存策略
- 安全漏洞：输入验证、权限控制、数据保护

**检查方法：**
- 静态代码分析
- 代码规范检查
- 安全扫描
- 性能分析

**职责边界：** 只做基础注释检查，详细注释检查由 `cs-doc` 负责

### 2. 注释检查（cs-doc）

**检查内容：**
- 注释完整性：类注释、方法注释、复杂逻辑注释、关键决策注释
- 注释清晰度：语言简洁、避免废话、更新及时、避免误导
- 注释规范性：格式规范、命名规范、位置规范、语言规范

**检查方法：**
- 注释完整性分析
- 注释质量评估
- 注释规范检查

**职责边界：** 专注于注释质量，不检查代码质量、性能、安全

### 3. 代码简化（cs-simplify）

**检查内容：**
- 代码冗余：重复代码、未使用的变量、冗余逻辑
- 复杂度优化：嵌套过深、条件复杂、循环冗长
- 重构机会：提取方法、合并条件、简化表达式

**检查方法：**
- 代码复杂度分析
- 重复代码检测
- 重构建议生成

**职责边界：** 专注于代码简化，不检查注释质量、功能验证

### 4. 代码分析（cs-analyze）

**检查内容：**
- 代码质量：命名规范、代码结构
- 潜在问题：内存泄漏、资源未释放、线程安全
- 改进建议：设计模式、架构优化、最佳实践

**检查方法：**
- 静态分析工具
- 代码质量指标
- 最佳实践检查

**职责边界：** 专注于代码分析，不检查代码简化、功能验证

### 5. 功能验证（function-validation）

**检查内容：**
- 功能正确性：是否符合需求、是否处理边界条件
- 最佳实践：是否遵循设计原则、是否使用合适的设计模式
- 代码质量：是否可测试、是否可维护、是否可扩展

**检查方法：**
- 单元测试验证
- 集成测试验证
- 功能测试验证

检查完成后，输出以下格式的检查报告：

```text

### 检查结果
```

## 直接读取 `skills/pua-escalation/SKILL.md` 的触发条件

出现任一情况，先读取 `skills/pua-escalation/SKILL.md` 获取解决方案，再回到质量检查流程：

- 你想跳过质量检查
- 你准备带着已知问题往下走
- 你在淡化检查发现的风险
- 你只想赶紧过关，不想修复问题
- 你打算用"后面再补"推迟关键问题

## 高压下怎么做

时间越紧，越不能缩质量检查。

正确做法是：

- 检查范围更聚焦
- 重要问题立即处理
- 一旦出现跳检查倾向，立即调 `pua-escalation`

## 红旗信号

- "这点东西不用检查"
- "这个后面再补"
- "先这样吧，差不多了"
- "这个太小，不值得检查"

## 压力升级

当遇到困难或失败时，读取 `skills/pua-escalation/SKILL.md` 获取具体的解决方案和指导。

## 底线

代码质量检查的成本，永远低于问题滚大后的成本；一旦开始想跳检查，就先调 `pua-escalation`。
