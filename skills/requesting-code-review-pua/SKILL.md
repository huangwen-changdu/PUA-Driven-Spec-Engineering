# Requesting Code Review PUA

## 概述

尽早发起评审，别让错误越滚越大。

**核心原则：** 早点审，早点修，不要把隐藏缺陷带到后面。出现想跳过 review、淡化风险、或只想赶进度时，直接调用 `pua-escalation`，再回到评审准备阶段继续推进。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了“有流程感”展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 确定评审范围
2. 提供准确的需求、任务或变更上下文
3. 同时检查正确性和质量
4. 区分严重、重要、次要问题
5. 判断是否需要调用 `pua-escalation`
6. 真实问题先修完，再往下走

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到 review 流程：

- 你想说“这点东西不用审”
- 你准备先带着已知问题往下走
- 你在淡化 review 发现的风险
- 你只想赶紧过关，不想补齐上下文
- 你打算用“后面再补”推迟关键问题

## 子 agent 约束

凡是派子 agent 做 review，prompt 必须带 PUA 质量约束：先查事实、别猜、输出证据、发现不一致要指出、不得空口完成。否则视为 review 子通道裸奔。

## 高压下怎么做

时间越紧，越不能缩 review。

正确做法是：

- 范围更聚焦
- 上下文更准确
- 重要问题立即处理
- 一旦出现跳 review 倾向，立即调 `pua-escalation`

## 红旗信号

- “这点东西不用审”
- “这个后面再补”
- “先这样吧，差不多了”

## 压力升级（内联保底）

优先调用 pua-escalation；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 pua-escalation 按 pua/references/methodology-router.md 切换链执行；无法加载时按内联切换动作表执行。

## 底线

评审的成本，永远低于错误滚大后的成本；一旦开始想跳 review，就先调 `pua-escalation`。
