# C# Analyze PUA

## 概述

分析 C# 代码质量，找出潜在问题和改进建议。

**核心原则：** 代码分析应该在开发过程中进行，不要等到最后才检查。出现想跳过分析、忽视问题、或只想赶进度时，直接调用 `pua-escalation`，再回到分析流程继续推进。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了"有流程感"展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 确定分析范围
2. 分析代码质量
3. 识别潜在问题
4. 生成改进建议
5. 评估代码质量
6. 判断是否需要调用 `pua-escalation`
7. 真实问题先修完，再往下走

## 检查内容

### 代码质量
- 命名规范：变量、方法、类的命名是否规范
- 注释质量：注释是否清晰、完整、有用
- 代码结构：代码组织是否合理、模块划分是否清晰

### 潜在问题
- 内存泄漏：未释放的资源、未关闭的连接
- 资源未释放：文件、数据库连接、网络连接
- 线程安全：多线程环境下的安全问题

### 改进建议
- 设计模式：是否可以使用合适的设计模式
- 架构优化：是否可以优化架构设计
- 最佳实践：是否遵循最佳实践

## 分析方法

### 1. 静态分析
```csharp
// 检查未使用的变量
int unusedVariable = 10; // 警告：未使用的变量

// 检查空值引用
string text = null;
int length = text.Length; // 错误：空值引用

// 检查资源释放
var stream = new FileStream("file.txt", FileMode.Open);
// 缺少 using 或 try-finally
```

### 2. 代码质量指标
- 圈复杂度：方法的复杂度
- 代码行数：方法的行数
- 参数数量：方法的参数数量

### 3. 最佳实践检查
- 异常处理：是否正确处理异常
- 资源管理：是否正确释放资源
- 线程安全：是否考虑线程安全

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到分析流程：

- 你想跳过代码分析
- 你准备忽视潜在问题
- 你在淡化分析发现的风险
- 你只想赶进度，不想修复问题
- 你打算用"后面再补"推迟关键问题

## 高压下怎么做

时间越紧，越不能缩代码分析。

正确做法是：

- 分析范围更聚焦
- 重要问题立即处理
- 一旦出现跳分析倾向，立即调 `pua-escalation`

## 红旗信号

- "这点东西不用分析"
- "这个后面再补"
- "先这样吧，差不多了"
- "这个太小，不值得分析"

## 压力升级（内联保底）

优先调用 pua-escalation；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 pua-escalation 按 pua/references/methodology-router.md 切换链执行；无法加载时按内联切换动作表执行。

## 底线

代码分析的成本，永远低于问题滚大后的成本；一旦开始想跳分析，就先调 `pua-escalation`。
