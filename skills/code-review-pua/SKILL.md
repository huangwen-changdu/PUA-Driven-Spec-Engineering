# Code Review PUA

## 概述

审查代码质量，发现潜在问题，优化代码性能，检查安全漏洞。

**核心原则：** 代码审查应该尽早进行，不要把隐藏缺陷带到后面。出现想跳过审查、淡化风险、或只想赶进度时，直接调用 `pua-escalation`，再回到审查流程继续推进。

## 第 0 步：自适应门禁

进入本 skill 后，先经过 `pua-gate` 并按档位执行：

- `G0/G1 PASS`：普通问题轻量快放，输出一行微标/轻标（不得完全静默）
- `G2 PASS`：多步骤或跨文件任务输出简版门禁
- `ESCALATE`：立即调用 `pua-escalation`，再回到本 skill
- `BLOCKED`：补齐输入或证据，不得继续

未经过 `pua-gate`，不得进入下面的主流程；但普通问题不得为了"有流程感"展开长门禁。

## 必经流程

0. 确认**本轮用户消息**已完成第 0 步门禁；不能复用上一轮 `pua-gate` 结果
1. 确定审查范围
2. 检查代码质量
3. 发现潜在问题
4. 优化代码性能
5. 检查安全漏洞
6. 汇总审查结果
7. 判断是否需要调用 `pua-escalation`
8. 真实问题先修完，再往下走

## 检查内容

### 代码质量
- 可读性：命名规范、代码结构
- 可维护性：模块化、低耦合、高内聚
- 可扩展性：接口设计、抽象层次、扩展点

### 注释质量
- 注释完整性：类注释、方法注释、复杂逻辑注释、关键决策注释
- 注释清晰度：语言简洁、避免废话、更新及时、避免误导
- 注释规范性：格式规范、命名规范、位置规范、语言规范

**注：** 注释质量的详细检查由 `cs-doc-pua` 技能负责，本技能只做基础检查。

### 潜在问题
- 逻辑错误：边界条件、异常处理、空值检查
- 性能问题：算法效率、资源使用、缓存策略
- 安全漏洞：输入验证、权限控制、数据保护

### 优化建议
- 重构机会：提取方法、合并条件、简化表达式
- 性能优化：算法优化、缓存策略、异步处理
- 架构改进：设计模式、模块划分、接口抽象

### 审查结果

   - 可读性：{评分}

## 直接调用 `pua-escalation` 的触发条件

出现任一情况，先调用 `pua-escalation`，再回到审查流程：

- 你想跳过代码审查
- 你准备带着已知问题往下走
- 你在淡化审查发现的风险
- 你只想赶紧过关，不想修复问题
- 你打算用"后面再补"推迟关键问题

## 高压下怎么做

时间越紧，越不能缩代码审查。

正确做法是：

- 审查范围更聚焦
- 重要问题立即处理
- 一旦出现跳审查倾向，立即调 `pua-escalation`

## 红旗信号

- "这点东西不用审查"
- "这个后面再补"
- "先这样吧，差不多了"
- "这个太小，不值得审查"

## 压力升级（内联保底）

优先调用 pua-escalation；无法加载时按以下规则就地执行：

| 失败次数 | 等级 | 强制动作 |
|---------|------|--------|
| 2 | E1 | 换本质不同的方案，不准换参数冒充 |
| 3 | E2 | 搜索 + 读上下文 + 列 3 个假设 |
| 4 | E3 | 7 项检查清单全部填完才继续 |
| 5+ | E4 | 拼命模式：升级搜索、验证、review |

**E3+ 检查清单**：逐字读失败信号？搜索过核心问题？读过原始上下文？假设用工具确认？试过反向假设？最小范围复现？换过工具/方法？

**方法论切换**：优先由 pua-escalation 按 pua/references/methodology-router.md 切换链执行；无法加载时按内联切换动作表执行。

## 底线

代码审查的成本，永远低于问题滚大后的成本；一旦开始想跳审查，就先调 `pua-escalation`。
