# PUA 压力引擎优化需求规范

## 概述
将 PUA 技能优化为方法论智能路由引擎，强化方法论在开发过程中的实时应用，清理其他 skill 中的压力升级废话，使 PUA 技能能够真实介入开发流程。

## 功能需求

### 1. 方法论智能路由（核心功能）
- **需求**：按任务类型自动选择最优方法论，真实引导 LLM 读取对应方法论文件
- **实现**：强化方法论路由表，确保开发过程中真实读取和应用方法论文件
- **效果**：开发过程中能够获得方法论文件的真实指导，提高开发效率和质量

**方法论文件路由**：
| 任务类型 | 推荐味道 | 方法论文件 | 核心方法 |
|---------|---------|-----------|---------|
| Debug/修 Bug | 🔴 华为 | `references/methodology-huawei.md` | RCA 根因分析 + 蓝军自攻击 |
| 构建新功能 | ⬛ Musk | `references/methodology-tesla.md` | The Algorithm: 质疑→删除→简化→加速→自动化 |
| 代码审查 | ⬜ Jobs | `references/methodology-apple.md` | 减法优先 + 像素级完美 + DRI |
| 调研/搜索 | ⚫ 百度 | `references/methodology-baidu.md` | 搜索是第一生产力 |
| 架构决策 | 🔶 Amazon | `references/methodology-amazon.md` | Working Backwards + 6-Pager |
| 性能优化 | 🟡 字节 | `references/methodology-bytedance.md` | A/B Test + 数据驱动 |
| 部署/运维 | 🟠 阿里 | `references/methodology-alibaba.md` | 定目标→追过程→拿结果闭环 |
| 任务模糊 | 🟠 阿里 | `references/methodology-alibaba.md` | 通用闭环（默认） |

**真实引导机制**：
- 接到任务后，先读取 `references/methodology-router.md` 确定任务类型
- 根据任务类型，引导 LLM 读取对应的方法论文件
- 在开发过程中，按照方法论文件的指导执行
- 失败时，引导 LLM 读取其他方法论文件

### 2. 统一压力升级引擎
- **需求**：将所有压力升级逻辑集中到 PUA 技能中
- **实现**：优化 `pua/SKILL.md`，使其成为统一的压力升级处理中心
- **效果**：其他 skill 不再需要重复压力升级逻辑

### 2. 清理其他 skill
- **需求**：删除其他 skill 中的压力升级废话
- **实现**：移除所有 skill 中的"压力升级（内联保底）"章节
- **效果**：减少重复内容，提高维护效率

### 3. 简化 PUA 技能
- **需求**：移除 PUA skill 中的请求接口和上报信息
- **实现**：删除 `pua/SKILL.md` 中的请求接口和上报相关内容
- **效果**：简化 PUA 技能，专注于核心功能

### 4. 真实介入机制
- **需求**：设计 PUA 技能真实介入开发流程的机制
- **实现**：设计监控、发现、介入处理的流程
- **效果**：PUA 技能能够真正参与开发过程

## 验收标准
- [ ] PUA 技能成为方法论智能路由引擎
- [ ] 开发过程中能够真实引导 LLM 读取对应方法论文件
- [ ] 按照方法论文件的指导执行开发任务
- [ ] PUA 技能成为统一的压力升级引擎
- [ ] 其他 skill 中的压力升级废话被删除
- [ ] PUA skill 中的请求接口和上报信息被移除
- [ ] PUA 技能能够真实介入开发流程
- [ ] 所有压力升级逻辑由 PUA 技能统一处理

## 非目标
- 不改变 PUA 的核心理念（压力、验证、闭环）
- 不破坏现有技能系统的基本架构
- 不删除必要的规则内容
- 不引入新的复杂机制

## 边界条件
- **边界情况1**：如果其他 skill 中的压力升级逻辑是必需的，如何处理？
  - **处理方式**：保留核心逻辑，但委托给 PUA 引擎处理
- **边界情况2**：如果 PUA 技能无法处理某些压力升级场景，如何处理？
  - **处理方式**：设计降级机制，确保不会破坏现有功能
