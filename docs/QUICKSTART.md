# 5 分钟快速开始

> **目标**：让你在 5 分钟内理解并开始使用 PUA-Driven Spec Engineering 流程。

## 核心理念

**一句话**：让 AI 编码助手更严谨、更可控，防止"说完成但没验证"、"跳过设计直接写代码"等问题。

**三个关键点**：
1. **自适应门禁**：普通问题轻量快放，高压问题走完整流程
2. **先设计后实现**：新功能必须先做方案对比，再写代码
3. **验证闭环**：说"完成"前必须有验证命令和真实输出

## 快速上手（3 种场景）

### 场景 1：简单修改（G0/G1）⏱️ 1-2 步

**适用**：改一行代码、解释代码、单文件小改、无风险信息查询

**怎么做**：直接说需求，AI 会自动判断并输出微标。

**示例**：
```
用户：帮我解释一下这个函数的作用
AI：🟠 PUA · 阿里味 · G0 · 先查事实再回答
   [解释内容]
```

### 场景 2：新功能（G2+）⏱️ 3-5 步

**适用**：新增功能、架构变更、模糊需求、真实业务需求

**怎么做**：说"我要做一个..."，AI 会引导你完成设计。

**示例**：
```
用户：我要给用户系统添加一个"收藏夹"功能
AI：🟠 PUA · 阿里味 · G2 · 约束：先理解再行动
   理解确认：我搜了用户系统代码，发现当前没有收藏相关功能...
   ⚠️ 不确认的风险：如果跳过设计，可能遗漏数据模型和接口设计
   任务类型：新功能
   成熟度：6/10 · 风险：R1
   下一步：brainstorming-pua（设计阶段）
   验收标准：用户可以收藏/取消收藏商品，收藏列表可查询
```

### 场景 3：紧急修复（/pua）⏱️ 2-4 步

**适用**：Bug 修复、紧急问题、用户不满、连续失败

**怎么做**：输入 `/pua`，AI 会进入高压模式快速定位问题。

**示例**：
```
用户：/pua 用户登录失败，错误日志显示...
AI：🟠 收到，登录失败还没用事实闭环，这叫自嗨。先查源码/日志/配置，用证据说话。
   PUA-GATE: ESCALATE · 来源：systematic-debugging-pua · 触发：用户不满
   [立即搜索代码、读取日志、定位根因]
```

## 决策速查表

| 你的需求 | 推荐路径 | 预计步骤 | 关键提示 |
|---------|---------|---------|---------|
| 改一行代码 | 直接说需求 → G0/G1 | 1-2 步 | AI 会自动判断影响面 |
| 解释代码 | 直接问 → G0 | 1 步 | 不需要走流程 |
| 新增功能 | 说需求 → AI引导设计 → 实现 | 3-5 步 | 必须先做设计 |
| 修复 Bug | 说问题 → AI定位 → 修复 | 2-4 步 | 用 `/pua` 进入高压模式 |
| 架构变更 | 说目标 → 完整设计 → 实现 | 5-8 步 | 必须走完整 OpenSpec 流程 |
| 模糊需求 | 说想法 → AI澄清 → 设计 → 实现 | 4-6 步 | AI 会逐项澄清 |

## 快速通道条件

满足以下**全部**条件时，可直接进入实现（跳过设计）：
1. 需求成熟度 ≥ 8
2. 变更风险 ≤ R1
3. 影响文件 ≤ 2 个
4. 用户明确说"我知道怎么做"
5. 无新功能/新模块/新API

## 回退命令

- `/back`：回到上一步
- `/restart`：重新开始当前阶段
- `/simplify`：切换到简化模式
- `/full`：切换到完整模式
- `/diagnose`：流程诊断

## 常见问题

### Q: 为什么 AI 要先搜索代码？
**A**: 为了确认影响面。AI 需要知道你的修改会影响哪些文件、模块、接口，防止"改了这里坏了那里"。

### Q: 为什么新功能必须先做设计？
**A**: 防止返工。直接写代码容易遗漏边界条件、接口设计、数据模型等问题，后期修改成本更高。

### Q: 什么是"验证闭环"？
**A**: 说"完成"前必须有验证命令和真实输出。不能说"应该可以了"，必须跑测试、看日志、确认结果。

### Q: 什么时候用 `/pua`？
**A**: 当你对 AI 的回答不满意、发现事实错误、连续失败、或需要紧急修复时。`/pua` 会让 AI 进入高压模式，优先搜索事实，快速定位问题。

### Q: 流程太复杂了，能简化吗？
**A**: 可以。满足"快速通道条件"时，AI 会自动跳过设计阶段。你也可以用 `/simplify` 命令切换到简化模式。
