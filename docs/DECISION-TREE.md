# 🌳 决策速查表

> **目标**：帮你快速判断应该走哪条路径，减少决策焦虑。

## 一、需求类型判断

### 1.1 你的需求是什么类型？

| 需求类型 | 特征 | 推荐路径 | 预计步骤 |
|---------|------|---------|---------|
| **简单修改** | 改一行代码、改配置、改文档 | G0/G1 直接执行 | 1-2 步 |
| **解释代码** | 问"这个函数做什么"、"这段代码逻辑" | G0 直接回答 | 1 步 |
| **Bug 修复** | 功能异常、报错、不符合预期 | G2+ 定位根因 → 修复 | 2-4 步 |
| **新功能** | 从无到有创建新功能 | G2+ 设计 → 实现 | 3-5 步 |
| **已有功能迭代** | 继续增强、修复、回滚或重构同一功能 | 先查 Feature Ledger → 新 change → 实现 | 3-5 步 |
| **架构变更** | 模块重构、接口变更、数据模型调整 | G2+ 完整设计 → 实现 | 5-8 步 |
| **模糊需求** | "用户需要..."、"希望实现..." | G2+ 澄清 → 设计 → 实现 | 4-6 步 |

### 1.2 快速判断流程图

```
你的需求是什么？
    │
    ├─ 改一行代码/配置/文档 → G0/G1 直接执行
    │
    ├─ 问代码逻辑/解释 → G0 直接回答
    │
    ├─ 修复 Bug → 说问题 → AI 定位 → 修复
    │
    ├─ 新增功能 → 说需求 → AI 引导设计 → 实现
    │
    ├─ 继续迭代已有功能 → 先查 `openspec/features/{feature-name}/README.md` → 新 change → 实现
    │
    ├─ 架构变更 → 说目标 → 完整设计 → 实现
    │
    └─ 模糊需求 → 说想法 → AI 澄清 → 设计 → 实现
```

## 二、风险等级判断

### 2.1 变更风险等级

| 风险等级 | 触发条件 | 流程要求 | 文档要求 |
|---------|---------|---------|---------|
| **R0 无风险** | 不涉及数据/资金/权限/核心流程 | 按成熟度档位走 | 无强制要求 |
| **R1 低风险** | 涉及 1 项，且为增量新增 | 实现前需代码搜索确认无冲突 | 无强制要求 |
| **R2 中风险** | 涉及 1 项且为修改/删除，或涉及 2 项及以上 | 必须进入 brainstorming-pua 做方案对比 | 必须落地设计文档 |
| **R3 高风险** | 涉及资金/权益计算，或涉及数据迁移/历史兼容 | ESCALATE + 必须完成影响面搜索 + 必须给出回滚方案 | 必须落地设计文档 + 回滚方案文档 |
| **R4 关键风险** | 涉及线上资金结算逻辑变更，或涉及生产数据迁移 | BLOCKED：必须完整影响面报告 + 回滚方案 + 测试验证证据 | 必须落地设计文档 + 回滚方案文档 + 影响面分析文档 |

### 2.2 高风险信号识别

命中以下**任一信号**，触发变更风险评估：

- **数据持久化变更**：新增/修改/删除数据库表、字段、索引、迁移脚本、ORM 映射
- **资金/权益计算变更**：涉及金额、折扣、积分、余额、分成、结算、退款、优惠等数值计算逻辑
- **权限/认证变更**：角色、权限、授权、鉴权、Token、登录态等变更
- **核心业务流程变更**：订单流程、支付流程、数据同步、定时任务核心链路
- **跨服务/跨模块接口变更**：API 契约变更、消息格式变更、共享数据结构变更
- **数据迁移/历史数据处理**：需要跑迁移脚本、清洗历史数据、兼容新旧格式

### 2.3 快速风险判断

```
你的变更涉及什么？
    │
    ├─ 不涉及任何高风险信号 → R0
    │
    ├─ 涉及 1 项，且为新增 → R1
    │
    ├─ 涉及 1 项且为修改/删除，或涉及 2 项及以上 → R2
    │
    ├─ 涉及资金/权益计算，或涉及数据迁移 → R3
    │
    └─ 涉及线上资金结算逻辑变更，或涉及生产数据迁移 → R4
```

## 三、成熟度判断

### 3.1 需求成熟度评分

| 维度 | 0 分（完全缺失） | 1 分（部分明确） | 2 分（充分明确） |
|------|----------------|----------------|----------------|
| **目标** | 只有一个方向，没有可交付的完成定义 | 有大致目标，但验收标准模糊 | 有明确可交付的完成定义 |
| **约束** | 没说边界、不做哪些、性能/兼容要求 | 有部分约束，但关键边界缺失 | 边界、排除项、非功能要求都明确 |
| **影响面** | 未知上下游、未知消费者 | 知道大致影响范围，细节不清 | 上下游、消费者、副作用都已识别 |
| **唯一性** | 可能有多种理解方式 | 主要理解方向一致，细节有歧义 | 只有一种合理理解 |
| **实现路径** | 无任何技术方案 | 有方向但有技术不确定 | 技术方案已验证或唯一 |

### 3.2 成熟度档位

| 成熟度总分 | 档位 | 行动 | 路由 |
|----------:|------|------|------|
| 0-2 | **G4 阻塞** | `BLOCKED`：需求几乎完全空白，必须先澄清核心意图 | 停止推进，问一个最小问题 |
| 3-4 | **G3 可见 PUA** | `ESCALATE`：需求严重不足，直接实现必然返工 | `brainstorming-pua` + `pua-escalation` |
| 5-6 | **G2 标准门禁** | 需要设计澄清后再进入实现 | `brainstorming-pua` |
| 7-8 | **G1 轻量门禁** | 需求基本清晰，但仍需确认剩余歧义 | 已确认实现步骤可进入 `writing-plans-pua`；真实业务需求必须进入 `brainstorming-pua` |
| 9-10 | **G0 微标快放** | 需求完全清晰，可直接实现 | 按任务类型走对应技能 |

### 3.3 快速成熟度判断

```
你的需求有多清晰？
    │
    ├─ 只有一个方向，没有完成定义 → 0-2 分 → G4 阻塞
    │
    ├─ 有大致目标，但验收标准模糊 → 3-4 分 → G3 可见 PUA
    │
    ├─ 需求基本清晰，但有部分歧义 → 5-6 分 → G2 标准门禁
    │
    ├─ 需求清晰，但需要确认细节 → 7-8 分 → G1 轻量门禁
    │
    └─ 需求完全清晰，可直接实现 → 9-10 分 → G0 微标快放
```

## 六、决策辅助工具

### 6.1 快速通道条件

满足以下**全部**条件时，可直接进入实现（跳过设计）：
1. 需求成熟度 ≥ 8
2. 变更风险 ≤ R1
3. 影响文件 ≤ 2 个
4. 用户明确说"我知道怎么做"
5. 无新功能/新模块/新API

### 6.2 强制设计条件

命中以下**任一**条件时，必须进入 brainstorming-pua：
1. 需求成熟度 ≤ 6
2. 新功能/新模块/新API/架构变更
3. 真实业务需求/用户故事/体验流程变更
4. 变更风险 ≥ R2
5. 存在多种可行方案需要对比
6. 影响面超过当前文件/模块
7. 存在任何业务边界或验收标准未确认

### 6.3 回退命令

- `/back`：回到上一步
- `/restart`：重新开始当前阶段
- `/simplify`：切换到简化模式
- `/full`：切换到完整模式
- `/diagnose`：流程诊断

## 七、常见问题

### Q: 为什么 AI 要先搜索代码？
**A**: 为了确认影响面。AI 需要知道你的修改会影响哪些文件、模块、接口，防止"改了这里坏了那里"。

### Q: 为什么新功能必须先做设计？
**A**: 防止返工。直接写代码容易遗漏边界条件、接口设计、数据模型等问题，后期修改成本更高。

### Q: 什么是"验证闭环"？
**A**: 说"完成"前必须有验证命令和真实输出。不能说"应该可以了"，必须跑测试、看日志、确认结果。

### Q: 什么时候用 `/pua`？
**A**: 当你对 AI 的回答不满意、发现事实错误、连续失败、或需要紧急修复时。`/pua` 会让 AI 进入高压模式，优先搜索事实，快速定位问题。

### Q: 流程太复杂了，能简化吗？
**A**: 可以。满足"快速通道条件"时，AI 会自动跳过设计阶段。你也可以用 `/simplify` 命令切换到简化模式。

---

> **记住**：决策速查表的目的是帮你快速判断应该走哪条路径，减少决策焦虑。AI 会根据你的需求自动判断，但了解这些规则可以让你更好地与 AI 协作。
